# UnrealMCP Capability Gaps

Tracking list of features the agent layer needed but the current MCP plugin does not expose.
Each entry is written as an actionable spec: what's missing, why it mattered in real work,
where to implement in the C++ plugin, and what the JSON shape should look like after the fix.

## Source of Findings

Compiled from a Demo walkthrough session (2026-04-17) against the TwistedBytes Interaction
System Demo (TBIA_Demo). Agent was using UnrealMCP to read Blueprint structure and asset
wiring as a substitute for opening the Editor UI. The gaps below blocked or forced heavy
filesystem/binary workarounds during that session.

---

## Tier 1 — Blocking Core Inspection (implement first)

These four gaps prevent the agent from answering basic questions like "what does this BP
inherit from, what's attached to it, and what's configured in its function graphs". Fix
these first; they are the floor for everything else.

### T1-1. Pin default values are not exposed

**Current behavior**
`HandleGetBlueprintGraphInfo` in
`Source/UnrealMCP/Private/Commands/UnrealMCPBlueprintNodeCommands.cpp` serializes pins at
lines 2874-2908. It:

- Only emits a pin entry if the pin has connections OR is an `exec` pin (the filter at
  line 2904-2908).
- Even when a pin is emitted, the JSON only includes `pin_name`, `direction`, `pin_type`,
  and `connected_to`. Default/literal values are dropped.

**Why this matters**

For the Demo walkthrough, a `Make TBIA Interaction Context` MakeStruct node holds all the
configuration a planner sets on an interactable — `Duration`, `ActiveType`, `Montage`,
`CooldownDuration`, etc. All of these are stored as pin default values on unconnected
input pins. The agent saw only the output exec pin, meaning the *content* of the
interactable's configuration was completely opaque through MCP.

Same problem for `Send Gameplay Event to Interacting Actor`'s `EventTag` pin — its literal
`FGameplayTag` was invisible, so the agent couldn't trace which ability consumed the event
without grepping source.

**Proposed change**

1. Change the pin filter to always emit pins that carry a default value, not just pins with
   connections or exec pins. Suggested rule: emit pins where any of
   `Pin->LinkedTo.Num() > 0`, `Pin->PinType.PinCategory == TEXT("exec")`,
   `!Pin->DefaultValue.IsEmpty()`, `Pin->DefaultObject != nullptr`, or
   `!Pin->DefaultTextValue.IsEmpty()`.
2. On each emitted pin, serialize the default values. `UEdGraphPin` provides:
   - `DefaultValue` (FString) — most literal values (primitives, tag strings, etc.)
   - `AutogeneratedDefaultValue` (FString) — fallback for zero values
   - `DefaultObject` (UObject*) — UObject references, assets, classes
   - `DefaultTextValue` (FText) — FText literals

Recommended JSON shape when a default is present:

```json
{
  "pin_name": "EventTag",
  "direction": "input",
  "pin_type": "struct",
  "pin_subtype": "GameplayTag",
  "connected_to": [],
  "default_value": "TBIA.Interaction.Animation.Cosmetic",
  "default_object_path": null
}
```

Include `pin_subtype` (from `Pin->PinType.PinSubCategory` / `PinSubCategoryObject`) because
a lot of default values are only interpretable with the subtype (e.g. `FGameplayTag` vs
`FName` are both rendered as strings).

**Acceptance test**

Query `BP_InteractionTrigger_Timed.GetInteractionContext`. The `Make TBIA Interaction
Context` node must show a non-zero `Duration` default and a non-empty `ActiveType`
default. Query `BP_ShieldBattery_Item.EventGraph`; the `Send Gameplay Event to Interacting
Actor` node must expose `EventTag = "TBIA.Interaction.Animation.Cosmetic"` (or whatever
value is actually set).

### T1-2. Blueprint parent class is not exposed

**Current behavior**

`get_blueprint_graph_info` returns `available_graphs` and `nodes`, but nothing about the
Blueprint itself — no parent class, no implemented interfaces, no generated class path.

**Why this matters**

To determine whether `BP_PlayerCharacter_ASC_OnPlayerState` inherits from
`ATBIA_GASCharacterBase` (invasive) or from plain `ACharacter` (non-invasive), the agent
had to fall back to `strings` on the uasset binary. This is a basic question — "what is
this Blueprint's parent class" — and should not require binary scraping.

**Proposed change**

Add a top-level `blueprint` object to the `get_blueprint_graph_info` response, or
(preferred) introduce a separate lightweight command `get_blueprint_info` that returns:

```json
{
  "blueprint_name": "BP_PlayerCharacter_ASC_OnPlayerState",
  "path": "/Game/Twistedbytes/InteractionsDemo/Blueprints/Core/BP_PlayerCharacter_ASC_OnPlayerState",
  "parent_class": "/Script/Engine.Character",
  "parent_class_short": "Character",
  "generated_class": "/Game/.../BP_PlayerCharacter_ASC_OnPlayerState.BP_PlayerCharacter_ASC_OnPlayerState_C",
  "implemented_interfaces": [
    "/Script/TBIA_Runtime.TBIA_Interactable",
    "/Game/.../BPI_CharacterInteractions.BPI_CharacterInteractions_C"
  ],
  "is_data_only": false,
  "blueprint_type": "Normal"
}
```

Source fields:
- `Blueprint->ParentClass->GetPathName()`
- `Blueprint->GeneratedClass->GetPathName()`
- `Blueprint->ImplementedInterfaces[i].Interface->GetPathName()`
- `FBlueprintEditorUtils::IsDataOnlyBlueprint(Blueprint)`
- `Blueprint->BlueprintType`

A separate command is cleaner than bolting this onto `get_blueprint_graph_info` because
callers who just want metadata should not have to pay the cost of serializing every graph.

**Acceptance test**

Query `BP_PlayerCharacter` → `parent_class == "TBIA_GASCharacterBase"`. Query
`BP_PlayerCharacter_ASC_OnPlayerState` → `parent_class == "Character"`. Both return the
`TBIA_Interactable` interface in `implemented_interfaces`.

### T1-3. Blueprint component tree (SCS) is not exposed

**Current behavior**

No command reads `Blueprint->SimpleConstructionScript`. The only way to tell a Blueprint
has a `UTBIA_InteractionComponent` attached is to look at the EventGraph's
`K2Node_ComponentBoundEvent` nodes and reverse-engineer.

**Why this matters**

The components on an Actor/Character Blueprint are *the* description of how it plugs into
plugins. A TBIA-integrated Character is defined by the presence of
`UTBIA_AbilitySystemComponent` and `UTBIA_InteractionComponent`. Without the ability to
enumerate these, the agent cannot tell which plugin integrations a Blueprint opts into.

**Proposed change**

Add a `get_blueprint_components` command (or include a `components` field in the
`get_blueprint_info` response from T1-2):

```json
{
  "components": [
    {
      "name": "TBIA_AbilitySystemComponent",
      "component_class": "/Script/TBIA_Runtime.TBIA_AbilitySystemComponent",
      "parent_component": null,
      "is_native": true,
      "property_overrides": {
        "bReplicates": true
      }
    },
    {
      "name": "Mesh",
      "component_class": "/Script/Engine.SkeletalMeshComponent",
      "parent_component": null,
      "is_native": true,
      "property_overrides": {
        "SkeletalMesh": "/Game/.../SK_Mannequin.SK_Mannequin"
      }
    }
  ]
}
```

Source:
- Native components: walk the `Blueprint->ParentClass` CDO's
  `TInlineComponentArray<UActorComponent>`, marking `is_native=true`.
- SCS-added components: walk `Blueprint->SimpleConstructionScript->GetAllNodes()`.
  Each `USCS_Node` exposes:
  - `GetVariableName()` for the component's variable name.
  - `ComponentClass` for the UClass.
  - `ComponentTemplate` for the template that holds property overrides.
  - `ChildNodes` for the attach tree (use `ParentComponentOrVariableName` for the relation).

**Property overrides** are the diff between the component template and its CDO. A minimal
first pass can dump the full template serialized-to-JSON via
`FJsonObjectConverter::UStructToJsonObject`; a later pass can diff against the CDO to
avoid noise.

**Acceptance test**

Query `BP_PlayerCharacter` → `components` includes both `TBIA_AbilitySystemComponent` and
`TBIA_InteractionComponent` with `is_native=true` (inherited from `ATBIA_GASCharacterBase`).
Query `BP_PlayerCharacter_ASC_OnPlayerState` → `components` includes
`TBIA_InteractionComponent` with `is_native=false` (added at BP level), and does *not*
include `TBIA_AbilitySystemComponent`.

### T1-4. Interface override graphs are missed by `get_blueprint_graph_info`

**Current behavior**

In `HandleGetBlueprintGraphInfo` (`UnrealMCPBlueprintNodeCommands.cpp:2801-2803`):

```cpp
SearchGraphs(Blueprint->UbergraphPages);
SearchGraphs(Blueprint->FunctionGraphs);
SearchGraphs(Blueprint->MacroGraphs);
```

Interface override graphs live in `Blueprint->ImplementedInterfaces[i].Graphs` when the
Blueprint's serialization uses that storage form. They are not searched.

**Why this matters**

`BP_PlayerCharacter_ASC_OnPlayerState` has 11 `ITBIA_Interactable` override graphs stored
under `ImplementedInterfaces[].Graphs` (`GetCharacterContext`, `GetInteractionContext`,
`ValidatePreInteract`, `IsAvailableForInteraction`, `IsBlockedForInteraction`,
`IsInteractionStillValid`, `GetPreInteractSyncType`, `GetPostInteractSyncType`,
`GetInteractingActors`, `CustomTraceForInteractActiveAbility`,
`CustomTraceForInteractPassiveAbility`). `get_blueprint_graph_info` reports *zero* of
these in `available_graphs`.

This silently corrupts any agent analysis like "override frequency across samples" — the
agent undercounts overrides on this BP and would reach wrong conclusions about planner
workflow complexity.

Note: the sibling BP `BP_PlayerCharacter` stores the same interface overrides in
`FunctionGraphs` instead, so they *are* visible. The storage form varies per BP for
reasons that aren't clear (possibly BP creation age / UE version at creation time).
Regardless, both storage forms must be searched.

**Proposed change**

Add one line to the graph search in `HandleGetBlueprintGraphInfo`:

```cpp
SearchGraphs(Blueprint->UbergraphPages);
SearchGraphs(Blueprint->FunctionGraphs);
SearchGraphs(Blueprint->MacroGraphs);
for (const FBPInterfaceDescription& Interface : Blueprint->ImplementedInterfaces)
{
    SearchGraphs(Interface.Graphs);
}
```

Include `"InterfaceName"` in the `available_graphs` output so callers can distinguish
interface overrides from plain functions — either by tagging each entry as an object
(breaking change) or by adding a parallel `interface_graphs` map.

Non-breaking shape:

```json
{
  "available_graphs": ["EventGraph", "UserConstructionScript", "GetInteractionContext"],
  "interface_graphs": {
    "TBIA_Interactable": ["GetCharacterContext", "GetInteractionContext", "ValidatePreInteract"],
    "BPI_CharacterInteractions": ["GetASCOwner", "PlaySound"]
  }
}
```

**Acceptance test**

Query `BP_PlayerCharacter_ASC_OnPlayerState`. The response must list all 11
`TBIA_Interactable` override graphs. Query
`BP_PlayerCharacter_ASC_OnPlayerState.GetInteractionContext` and get back the node
contents.

## Tier 2 — Commonly Needed (implement second)

### T2-5. Blueprint CDO / Default values are not exposed

**Current behavior**

No command reads the Blueprint's Class Default Object. Every property visible in the
Details panel (outside the SCS component tree) is opaque.

**Why this matters**

A lot of the actual configuration of Blueprint-backed assets lives on the CDO, not in
graphs:

- `BP_PlayerCharacter`'s `GrantAbilitiesOnStart` array — this is what decides which
  abilities the player can consume.
- `BP_FirstPersonGameMode`'s `DefaultPawnClass`, `PlayerStateClass`, `HUDClass`.
- `BP_GAInteractAnimationCosmetic`'s `AbilityTriggers` array — a BP subclass of a GAS
  ability might override the trigger tag set by the C++ constructor.
- Component-property overrides that aren't covered by the SCS template diff in T1-3.

Without this, "what is this BP actually configured to do" is only half-answerable.

**Proposed change**

Add a `get_blueprint_defaults` command:

```
{"type": "get_blueprint_defaults", "params": {"blueprint_name": "BP_PlayerCharacter"}}
```

Response: walk the CDO (`Blueprint->GeneratedClass->GetDefaultObject()`) using
`FJsonObjectConverter::UStructToJsonObject` over the UObject's properties. For array/map
properties, recurse. For soft object references, emit the path. For hard references, emit
the full object path.

Optional `path` parameter to fetch a specific property subtree (e.g.
`"GrantAbilitiesOnStart"` or `"CharacterMovement.MaxWalkSpeed"`) to avoid dumping huge
trees when only one field is of interest.

**Acceptance test**

Query `BP_PlayerCharacter` → `GrantAbilitiesOnStart` array is non-empty and contains
paths like `/Game/.../BP_GAInteractPassive`. Query `BP_FirstPersonGameMode` →
`DefaultPawnClass` equals the expected Player BP.

### T2-6. DataAsset contents cannot be read

**Current behavior**

No command to read `UDataAsset` (or `UPrimaryDataAsset`) contents. This is a sibling
capability to T2-5 (both read UObject properties), but scoped to DataAssets on disk rather
than Blueprint CDOs.

**Why this matters**

`GASEnhancedInputMapping` is a `UTBIA_EnhancedInputMapping` DataAsset that holds the
`AbilityMapping[]` array — each entry a tuple of `InputAction`, `GameplayTag`, and
`AbilityClass`. This is the *central* configuration for how a player's key presses
translate to ability activations in TBIA. The Demo has one such DataAsset; real projects
will have many. Without a way to read them, the input pipeline stays opaque.

Generalizes to all `UDataAsset` content in the project.

**Proposed change**

Add `get_data_asset`:

```
{"type": "get_data_asset", "params": {"path": "/Game/.../GASEnhancedInputMapping"}}
```

Implementation: `StaticLoadObject<UDataAsset>` (or `LoadObject`) then
`FJsonObjectConverter::UStructToJsonObject` on the object's property container. Reject
(return error) if the asset is not a `UDataAsset` or `UPrimaryDataAsset`, or accept any
`UObject` asset via a second parameter `allow_any_object=true` for non-DataAsset configs
that happen to be serialized as assets (less common but real).

**Acceptance test**

Query `GASEnhancedInputMapping` → response contains `AbilityMapping` array with entries
exposing `InputAction`, `InputTag`, and `AbilityClass` fields.

### T2-7. No asset-registry queries

**Current behavior**

The agent cannot enumerate assets via MCP. Present commands are either actor-scoped
(`get_actors_in_level`, `find_actors_by_name`) or Blueprint-name-scoped (requires knowing
the name up front). `UnrealMCPCommonUtils.cpp` already imports `AssetRegistryModule` for
other purposes but does not expose a listing command.

**Why this matters**

During the Demo walkthrough, the agent repeatedly had to `find` on the filesystem to
discover assets — `find Content -name "BP_*"` or `find Content -name "DA_*"`. This is
slow, doesn't know asset class, and crosses the MCP boundary into raw shell.

**Proposed change**

Add `find_assets`:

```
{
  "type": "find_assets",
  "params": {
    "path": "/Game/Twistedbytes/InteractionsDemo",    // optional; default /Game
    "class_name": "Blueprint",                         // optional; short name or path
    "recursive": true,                                 // default true
    "name_pattern": "BP_Interactable*"                 // optional glob, case-insensitive
  }
}
```

Response:

```json
{
  "assets": [
    {
      "name": "BP_InteractableStaticMeshBase",
      "path": "/Game/.../Abstract/BP_InteractableStaticMeshBase",
      "class": "Blueprint",
      "parent_class": "TBIA_GASActorBase"    // bonus, if cheap
    }
  ],
  "total_count": 17
}
```

Implementation: `FAssetRegistryModule::Get().GetAssets(FARFilter)`. For Blueprints, the
parent class can be read from the asset's tags (`ParentClass` tag) without loading the
asset.

**Acceptance test**

Query `{"class_name": "Blueprint", "path": "/Game/Twistedbytes"}` returns the full list of
Demo Blueprints without needing to grep the filesystem.

## Tier 3 — Nice To Have (deferred until Tier 1+2 land)

These are workarounds-are-acceptable items. Recorded for future pickup.

### T3-8. Reverse-lookup: abilities by trigger tag

Given a `FGameplayTag`, find all `UGameplayAbility` subclasses whose `AbilityTriggers`
contains that tag. Current workaround: grep the source tree.

Implementation sketch: iterate `UGameplayAbility` subclasses via asset registry +
`UClass::GetChildren()`, inspect each CDO's `AbilityTriggers`.

### T3-9. Gameplay Tag registry query

Enumerate all registered tags, optionally filtered by prefix. Current workaround: grep
`UE_DEFINE_GAMEPLAY_TAG` and `DefaultGameplayTags.ini`.

Implementation: `UGameplayTagsManager::Get().RequestAllGameplayTags()`.

### T3-10. List Blueprint variables

Enumerate a BP's user-defined variables (`Blueprint->NewVariables`) with type and default
values. Distinct from T2-5 which reads the full CDO; this lists only BP-authored
variables. Useful to understand what state a BP carries without dumping everything.

### T3-11. K2Node_Message implementation follow

Given a `K2Node_Message` node in a graph (e.g. "Target is BPI Character Interactions"),
return the list of Blueprints implementing that interface and the path to each
implementation's override graph.

Implementation: asset registry filter for BPs whose `ImplementedInterfaces` contains the
interface, for each BP locate the override graph by name.

### T3-12. Level / GameMode metadata

Return the current level's `GameMode`, `DefaultPawnClass`, `PlayerStateClass`, etc. —
either at the level level or via the effective GameMode chain
(`World Settings -> GameMode Override -> Project default`). Most of this is reachable via
T2-5 on the relevant Blueprints if you know which Blueprint to query; T3-12 is
discoverability sugar on top.

## Implementation Order

1. **Batch A (Tier 1)** — T1-1, T1-2, T1-3, T1-4. After this batch: basic Blueprint
   introspection ("what is this BP, what does it inherit, what's on it, what graphs does
   it have") is complete via MCP.
2. **Batch B (Tier 2)** — T2-5, T2-6, T2-7. After this batch: Details-panel-level
   configuration and asset discoverability are complete.
3. **Batch C (Tier 3)** — deferred.

Each batch should land as a single commit (or small set of commits) with:

- C++ handler + metadata updated in `Source/UnrealMCP/Private/Commands/`.
- Python tool layer updated under `Python/tools/`.
- `Docs/commands.md` updated.
- `Docs/Progress.md` updated with a dated entry.
- Live smoke test against at least one consumer project (TBIA_Demo is the first).

## Acceptance Criteria (meta)

Post-Tier-1, the following workflow must succeed end-to-end via MCP without filesystem
or `strings` fallback:

> Given a Blueprint name, determine its parent class, implemented interfaces, attached
> components, and for each of its function/interface graphs dump the nodes *and* each
> node's literal pin values.

Post-Tier-2, extend to:

> Given any asset path (Blueprint, DataAsset, GameMode), return the CDO / content, and
> enumerate all Blueprints under a folder by class.
